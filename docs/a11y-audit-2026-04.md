# TaskApp accessibility audit — April 2026

**Status:** snapshot report. No code fixes applied here — the fixes land
as separate PRs once we pick the top N issues.

## Method

1. `python3 tools/a11y_rn_filter.py mobile/` — wraps `.claude/skills/a11y-audit/scripts/a11y_scanner.py` and drops rule IDs that target HTML constructs React Native doesn't have (landmarks, headings, form `<label>`s, `<img alt>`, ARIA, `<a>`, `<table>`, `<video>`/`<audio>`). 74 raw findings → **0 findings that apply to React Native**. The scanner as-is is the wrong tool for an RN codebase; see *Follow-ups* below for what to replace it with.
2. `python3 .claude/skills/a11y-audit/scripts/contrast_checker.py <fg> <bg> --json` — swept the semantic tokens in `mobile/lib/colors.ts` against the three surface colors used across the app (`#ffffff`, `#f5f6fa`, `#e8f0fe`). Ratios below are WCAG 2.2.

## Filtered scanner findings

None. After filtering RN-incompatible rules the scanner has no remaining
signal against this codebase. That's not a clean bill of health — it
means the scanner isn't looking at the things that actually matter for
RN (accessibilityLabel coverage, tap-target sizing, accessibilityRole
correctness). See *Follow-ups*.

## Top recommendations

Keep these scoped — a one-PR sweep can land the first three and make a
real dent.

1. **Darken `textMuted` from `#999` to `#595959`** (~7:1 on white, AA
   Normal on all three backgrounds). Used everywhere for metadata
   (folder subtitles, task meta rows, "last Nd ago"). The UX review
   flagged this as 2.85:1 already. Global find/replace on the token is
   one PR.
2. **Stop using bare `#999` / `#888` in StyleSheets** as body text
   color; they bypass the token. `grep -rn "'#999'\|'#888'" mobile/app
   mobile/components` before the sweep to catch strays.
3. **`primary` as body text on white only**. On `#e8f0fe` (our tinted
   button / chip background) it hits 3.93:1 — AA Large passes but
   not Normal. OK for short button labels, not OK for sentence-length
   copy. Add a palette comment warning.
4. **`warning` / `success` / `danger` are NOT body-text colors.** All
   fail AA Normal on every surface. Current usage is mixed — some as
   icons (fine), some as 11–13 px text inside pills (fail). Either:
   - (a) Reserve for icons + big labels, use `text` for copy alongside.
   - (b) Introduce `warningText` / `successText` / `dangerText` tokens
     at AA-Normal strength (e.g. `#a05a00`, `#1e7c42`, `#a52a1a`).
5. **`textFaint`, `accent`, `warningSoft`** are decorative only. Never
   use for text. Add a prominent comment in `colors.ts`.

## What's NOT checked here

The upstream scanner is HTML-shaped; the checks that matter for RN
aren't in it. For a real RN a11y pass we need:

- **`accessibilityLabel` coverage** on `<Pressable>`, `<TouchableOpacity>`,
  and `<TextInput>`. A simple TypeScript-AST walker (ts-morph or
  `@typescript-eslint/parser`) can flag interactive elements that are
  missing both `accessibilityLabel` and a `<Text>` child. Candidate for
  a new `tools/a11y_rn_ast.py`.
- **44 × 44 tap targets.** StyleSheet entries that give width/height
  < 44 or rely on `padding: 6` around a 16–18 px icon. The
  `workout/[routineId]` reorder chevrons were caught manually — a
  static check would find the rest.
- **`accessibilityRole` correctness.** Native iOS/Android screen
  readers benefit from explicit roles on custom `<Pressable>` buttons.
- **Dynamic Type / font scaling.** Nothing in the repo disables
  `allowFontScaling`, which is good, but a spot check on the tab bar
  and session banner at 200 % font scale should confirm nothing
  clips.
